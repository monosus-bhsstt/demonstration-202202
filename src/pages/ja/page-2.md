## React は何が良いのか

現在の WEB アプリケーションにおいて React で実現できないことはないという前提に立ちつつ、静的なコーディングと比較した場合に特に顕著な優位性は、

- ページレイヤーではなくコンポーネントレイヤーで開発できる
- サーバサイド側とクライアント側の実装をコンポーネント単位で実現できる
- ロジック・スタイル・マークアップをコンポーネントレベルでカプセル化できる
- TypeScript を用いて型安全にコンポーネント管理できる
- 外部データを簡易に読み込める

例えば、

- テキストリンク要素に[PDF]といプロパティ（属性）をもたせると PDF アイコンが挿入されると同時に外部リンクが設定される
- 画像要素にはサイズ属性が自動で記述されるようにできる。alt 属性がない場合はエラーを投げるようにできる
- Youtube の動画 ID を引数に渡すだけでサムネイル付きの動画リンクを生成する
- クリック前にクリック先のページをローディングしておき高速に遷移させる
- テキストコンテンツは json として管理し p タグ等へ引数として挿入させる 等など。。。

もちろん他のフレームワークでも同一のことは実現できます。
他フレームワークと比較した React の優位点としては

- TypeScript への親和性
- OSS ライブラリの充実度
- 導入数によるナレッジの多さ、開発者人口

があります。

## React は最高なのか

React には大きな問題点があります。それは **初期表示のパフォーマンスに影響がでてしまう**というものです。

React で作られたアプリケーションは javascript の実行が完了してから表示されます。実装の設計が最適化されていない場合、コンテンツ量やページの複雑制によって初期レンダリングが遅くなってしまいます。よくあるローディング画面などを思い出してみてください。

```html
<html>
  <head>
    <title>sample</title>
  </head>
  <body>
    <div id="app"></div>
    <script src="/js/bundle.js"></script>
  </body>
</html>
<!-- よくあるReact App。htmlはjavascriptの実行のためだけに存在している -->
```

![No pre-rendring](https://nextjs.org/static/images/learn/data-fetching/no-pre-rendering.png)

ふんだんにユーザーインタラクションさせるようなアプリケーションであればよいのですが、マーケティングサイトやブログサイトでは効果的ではありません。

この問題を解決するために SSG（Static Site Generator）を用いて、あらかじめページ表示に必要なデータを取得し、各ページごとに静的な HTML ファイルを出力しておく。そしてサーバーへのリクエスト時にはこの HTML ファイルを返却する。という方法をとることが望ましいです。

![With pre-rendring](https://nextjs.org/static/images/learn/data-fetching/pre-rendering.png)

SSG の他に SSR（Server Side Rendering）や SSI を用いたページ管理方法もありますが、サイトパフォーマンスの視点からは、以下の質問に対して「No」と答えられない限りは SSG を採用すべきです。

> Q. ユーザのリクエストより先にこのページを Pre-rendering することができますか？
